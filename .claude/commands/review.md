# Code Review

審查本次對話中所做的所有程式碼變更，產出結構化的 review 報告。

## 步驟

### 1. 收集變更範圍

透過 `git log` 和 `git diff` 確認本次對話產生的所有 commit 與變更檔案。

### 2. 逐檔審查

對每個變更檔案進行以下檢查：

#### Bug 與正確性
- 邏輯錯誤、邊界條件、null 風險
- SQL 注入、未跳脫的使用者輸入
- 回傳值約定是否正確（create/save/delete 失敗回傳 null）
- Plugin 生命週期是否完整（allowTypes、toSqlString、updateValue）

#### 開發規範
- `.claude/rules/` 下所有規範（php、git）
- 命名慣例（private `$_` 前綴、方法 camelCase、常數 CONSTANT_CASE）
- PHP 7.4 相容性（禁止 PHP 8.0+ 語法）
- 程式碼風格一致性（與同檔案既有程式碼比對）

#### API 一致性
- Model 靜態方法與 Builder 實例方法的參數簽名是否同步
- 新增/修改的 public 方法是否與既有 API 風格一致
- 別名方法（如 `in` / `whereIn`）是否同步更新

#### 文件同步
- Guide/ 文件是否需要更新
- CLAUDE.md 的 API 速查表是否需要同步

#### 測試覆蓋
- 變更的功能是否有對應 Test/ 測試
- 現有測試是否仍能通過

#### 優化建議
- 可簡化或合併的程式碼
- 效能改善空間（不必要的查詢、重複計算）
- 可讀性改善

### 3. 產出報告

以表格形式整理每個檢查維度的結果：

```
## Review 結果

### [功能名稱 1]

| 檢查項目 | 結果 |
|----------|------|
| ... | ... |

### [功能名稱 2]
...

## 結論

- 是否有 bug 需要修復
- 是否有規範違反需要調整
- 優化建議（非必要，列出供參考）
```

### 4. 修復問題（若有）

若發現需要修復的問題：
1. 說明問題與修復方案
2. 詢問是否要立即修復
3. 修復後重新驗證

## 注意事項

- 審查範圍僅限本次對話的變更，不擴大到未修改的程式碼
- 優化建議只提非必要的改善，不強制執行
- 規範以 `.claude/rules/` 為準，不自行發明規則
